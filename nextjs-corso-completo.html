Setup e Primo Progetto

Installazione

Assicurati di avere Node.js installato (versione 18.17 o superiore).

node --version # Verifica versione Node.js

Creare un Progetto Next.js

Il modo più semplice è usare create-next-app:

npx create-next-app@latest my-first-nextjs-app

Durante l'installazione ti verranno fatte alcune domande:

✔ Would you like to use TypeScript? … No / Yes
✔ Would you like to use ESLint? … No / Yes
✔ Would you like to use Tailwind CSS? … No / Yes
✔ Would you like to use `src/` directory? … No / Yes
✔ Would you like to use App Router? … No / Yes  ← SCEGLI YES!
✔ Would you like to customize the default import alias? … No / Yes
💡 Configurazione Consigliata
  • TypeScript: No (per iniziare, aggiungi dopo)
  • ESLint: Yes
  • Tailwind: Yes (facilita lo styling)
  • src/ directory: No
  • App Router: YES! (importante)

Avviare il Server di Sviluppo

cd my-first-nextjs-app npm run dev

Apri il browser su http://localhost:3000 - vedrai la pagina di benvenuto!

Struttura del Progetto

Ecco cosa trovi nella cartella del progetto:

my-first-nextjs-app/
├── app/                    ← Il cuore dell'applicazione
│   ├── layout.js          ← Layout principale
│   ├── page.js            ← Homepage (/)
│   └── globals.css        ← Stili globali
├── public/                ← File statici (immagini, favicon)
├── node_modules/          ← Dipendenze
├── package.json           ← Configurazione npm
└── next.config.js         ← Configurazione Next.js
📁 La Cartella app/

Con l'App Router, tutto vive in app/. Ogni file page.js diventa automaticamente una route accessibile nel browser.

La Tua Prima Pagina

Apri app/page.js e sostituisci tutto con questo:

export default function Home() {
  return (
    <main style={{ padding: '2rem' }}>
      <h1>La Mia Prima App Next.js! 🚀</h1>
      <p>Sto imparando Next.js e sta andando alla grande!</p>
    </main>
  )
}

Salva il file - il browser si aggiornerà automaticamente! Questo è l'Hot Module Replacement in azione.

Creare una Seconda Pagina

Per creare la route /about:

  1. Crea una cartella app/about/
  2. Dentro crea page.js
// app/about/page.js
export default function About() {
  return (
    <main style={{ padding: '2rem' }}>
      <h1>Chi Sono</h1>
      <p>Sono uno sviluppatore che sta imparando Next.js!</p>
    </main>
  )
}

Vai su http://localhost:3000/about - funziona! ✨

🗂️ Routing File-Based

Non c'è bisogno di configurare route manualmente. La struttura delle cartelle in app/ diventa automaticamente il routing dell'app:

  • app/page.js → /
  • app/about/page.js → /about
  • app/blog/page.js → /blog

Navigazione tra Pagine

Per navigare tra pagine, usa il componente Link:

// app/page.js
import Link from 'next/link'

export default function Home() {
  return (
    <main style={{ padding: '2rem' }}>
      <h1>Homepage</h1>
      <Link href="/about">Vai alla pagina About</Link>
    </main>
  )
}
⚡ Perché Link invece di <a>?

<Link> fa navigazione client-side senza ricaricare la pagina. È più veloce e mantiene lo stato dell'app. Next.js fa anche prefetching automatico dei link visibili!

Routing File-Based

Convenzioni di Naming

Next.js usa nomi di file speciali per definire le route:

File Scopo
page.js Definisce una pagina accessibile via URL
layout.js Layout condiviso per pagine figlie
loading.js UI di caricamento
error.js UI per gestire errori
not-found.js Pagina 404 personalizzata

Route Dinamiche

Per creare route con parametri dinamici, usa le parentesi quadre [param]:

app/
├── blog/
│   ├── page.js           → /blog
│   └── [slug]/
│       └── page.js       → /blog/qualsiasi-cosa
// app/blog/[slug]/page.js
export default function BlogPost({ params }) {
  return (
    <div>
      <h1>Post: {params.slug}</h1>
    </div>
  )
}

Esempi di URL che matchano:

  • /blog/primo-post → params.slug = "primo-post"
  • /blog/imparare-nextjs → params.slug = "imparare-nextjs"

Route Dinamiche Annidate

Puoi avere più parametri dinamici:

app/
└── shop/
    └── [category]/
        └── [product]/
            └── page.js   → /shop/[category]/[product]
// app/shop/[category]/[product]/page.js
export default function Product({ params }) {
  return (
    <div>
      <h1>Categoria: {params.category}</h1>
      <h2>Prodotto: {params.product}</h2>
    </div>
  )
}

// URL: /shop/elettronica/laptop
// params = { category: 'elettronica', product: 'laptop' }

Catch-All Routes

Per catturare più segmenti di URL, usa [...slug]:

app/
└── docs/
    └── [...slug]/
        └── page.js       → /docs/a/b/c/d/e
// app/docs/[...slug]/page.js
export default function Docs({ params }) {
  return (
    <div>
      <h1>Documenti</h1>
      <p>Percorso: {params.slug.join('/')}</p>
    </div>
  )
}

// URL: /docs/api/users/create
// params.slug = ['api', 'users', 'create']

Route Groups

Organizza le route senza influenzare l'URL usando parentesi (nome):

app/
├── (marketing)/        ← Non appare nell'URL!
│   ├── about/
│   │   └── page.js    → /about (non /marketing/about)
│   └── contact/
│       └── page.js    → /contact
└── (shop)/
    └── products/
        └── page.js    → /products
🎨 Usa i Route Groups per:
  • Organizzare il codice senza cambiare URL
  • Applicare layout diversi a sezioni diverse
  • Dividere l'app in aree logiche (admin, public, ecc.)

Parallel Routes

Renderizza più pagine contemporaneamente usando slot @nome:

app/
├── layout.js
├── page.js
├── @team/
│   └── page.js
└── @analytics/
    └── page.js
// app/layout.js
export default function Layout({ children, team, analytics }) {
  return (
    <div>
      <header>Header</header>
      {children}
      <div style={{ display: 'grid', gridTemplateColumns: '1fr 1fr' }}>
        {team}
        {analytics}
      </div>
    </div>
  )
}

Intercepting Routes

Intercetta una route per mostrarla in modo diverso (es. modal):

app/
├── feed/
│   └── page.js
├── photo/
│   └── [id]/
│       └── page.js        → /photo/123 (full page)
└── (..)photo/             ← Intercetta /photo/*
    └── [id]/
        └── page.js        → Mostra come modal
🎭 Intercepting Routes

Perfetto per aprire contenuti in modal quando sei già in una pagina, ma mostrare la pagina completa se arrivi dall'esterno (es. Instagram-style photo viewer).

Crea un blog con:

  1. Pagina lista post (/blog)
  2. Pagina post dinamica (/blog/[slug])
  3. Array di post hardcoded con title, slug, content
  4. Link dalla lista ai singoli post
  5. Mostra il contenuto del post basato sullo slug

Bonus: Aggiungi una pagina 404 personalizzata se lo slug non esiste

Pages Router vs App Router

Due Paradigmi Diversi

Next.js ha due sistemi di routing. È importante capire le differenze:

Aspetto Pages Router App Router
Introdotto Next.js 1 (2016) Next.js 13 (2022)
Cartella pages/ app/
Componenti Default Client Components Server Components
Data Fetching getServerSideProps, getStaticProps async/await nei componenti
Layouts Tramite _app.js layout.js nativo
Stato Stabile Raccomandato (futuro)

Pages Router (Legacy)

pages/
├── index.js              → /
├── about.js              → /about
├── blog/
│   ├── index.js         → /blog
│   └── [slug].js        → /blog/:slug
└── _app.js              → Wrapper globale
// pages/blog/[slug].js - PAGES ROUTER
export default function BlogPost({ post }) {
  return <article>{post.title}</article>
}

// Data fetching - funzione separata!
export async function getStaticProps({ params }) {
  const post = await fetch(`/api/posts/${params.slug}`)
  return { props: { post } }
}

App Router (Moderno)

app/
├── page.js              → /
├── about/
│   └── page.js          → /about
├── blog/
│   ├── page.js          → /blog
│   └── [slug]/
│       └── page.js      → /blog/:slug
└── layout.js            → Layout nativo
// app/blog/[slug]/page.js - APP ROUTER
export default async function BlogPost({ params }) {
  // Fetch direttamente nel componente!
  const post = await fetch(`/api/posts/${params.slug}`)
  const data = await post.json()
  
  return <article>{data.title}</article>
}
🆕 Differenza Chiave

Con l'App Router, puoi fare fetch direttamente nei componenti. Non servono più funzioni separate come getStaticProps. I componenti possono essere async!

Vantaggi dell'App Router

  • Server Components di default - meno JavaScript al client
  • Streaming - parti della pagina caricano progressivamente
  • Layouts nativi - facili da annidare e riutilizzare
  • Data fetching semplificato - async/await ovunque
  • Parallel Routes - renderizza più pagine insieme
  • Intercepting Routes - modal e overlay sofisticati

Quando Usare Cosa?

✅ Usa App Router se:
  • Stai iniziando un nuovo progetto
  • Vuoi le performance migliori
  • Vuoi le feature più moderne
  • Sei disposto a imparare un nuovo paradigma
⚠️ Usa Pages Router se:
  • Hai un progetto esistente grande
  • Usi librerie che non supportano Server Components
  • Il team non è pronto per il cambiamento
  • Hai bisogno di massima stabilità

Migrazione Graduale

Puoi usare entrambi insieme! Next.js supporta pages/ e app/ nello stesso progetto:

my-app/
├── pages/              ← Route legacy
│   └── old-page.js
└── app/                ← Nuove route
    └── new-page/
        └── page.js

Le route in app/ hanno precedenza su quelle in pages/.

Confronto Codice

Metadata (SEO)

// PAGES ROUTER
import Head from 'next/head'

export default function Page() {
  return (
    <>
      <Head>
        <title>Il mio titolo</title>
        <meta name="description" content="..." />
      </Head>
      <div>Contenuto</div>
    </>
  )
}
// APP ROUTER
export const metadata = {
  title: 'Il mio titolo',
  description: '...'
}

export default function Page() {
  return <div>Contenuto</div>
}

Loading States

// PAGES ROUTER - manuale
import { useState, useEffect } from 'react'

export default function Page() {
  const [loading, setLoading] = useState(true)
  
  useEffect(() => {
    // fetch...
    setLoading(false)
  }, [])
  
  if (loading) return <div>Loading...</div>
  return <div>Contenuto</div>
}
// APP ROUTER - automatico con loading.js
// app/page.js
export default async function Page() {
  const data = await fetch('...')
  return <div>Contenuto</div>
}

// app/loading.js - mostrato automaticamente!
export default function Loading() {
  return <div>Loading...</div>
}

Questo corso si concentra sull'App Router perché è il futuro di Next.js. Tutto ciò che impari qui funzionerà per anni. Pages Router rimarrà supportato ma senza nuove feature.

Strategie di Rendering

I Tre Pilastri del Rendering

Next.js ti dà 3 modi per renderizzare le pagine. Ognuno ha i suoi casi d'uso:

1. Static Site Generation (SSG)

Quando: Build time (durante npm run build)
Cosa: HTML generato una volta, riutilizzato per ogni richiesta
Ideale per: Contenuti che non cambiano spesso

// app/about/page.js
export default function About() {
  return <h1>Chi Siamo</h1>
}

// Questa pagina è STATICA per default!
// Next.js genera about.html al build
✅ Vantaggi SSG
  • Performance massime - HTML pre-generato
  • Scalabilità - servibile da CDN
  • SEO perfetto - crawler vedono subito tutto
  • Costi bassi - solo file statici

2. Server-Side Rendering (SSR)

Quando: Ad ogni richiesta
Cosa: HTML generato on-demand sul server
Ideale per: Dati che cambiano frequentemente o personalizzati

// app/dashboard/page.js
async function getUserData() {
  // Questa funzione gira SUL SERVER ad ogni richiesta
  const res = await fetch('https://api.example.com/user', {
    cache: 'no-store' // ← Disabilita il cache = SSR!
  })
  return res.json()
}

export default async function Dashboard() {
  const user = await getUserData()
  
  return (
    <div>
      <h1>Benvenuto, {user.name}!</h1>
      <p>Ultimo accesso: {user.lastLogin}</p>
    </div>
  )
}
🔑 Cache Control

In App Router, Next.js decide automaticamente SSG o SSR basandosi su come fai fetch:

  • fetch(url) → cached (SSG)
  • fetch(url, {`{ cache: 'no-store' }`}) → non cached (SSR)
  • fetch(url, {`{ next: { revalidate: 60 } }`}) → ISR (vedi sotto)

3. Incremental Static Regeneration (ISR)

Quando: Build + revalidation periodica
Cosa: Statico ma si aggiorna ogni N secondi
Ideale per: Il meglio di SSG e SSR

// app/blog/page.js
async function getPosts() {
  const res = await fetch('https://api.example.com/posts', {
    next: { revalidate: 3600 } // ← Revalida ogni ora!
  })
  return res.json()
}

export default async function Blog() {
  const posts = await getPosts()
  
  return (
    <div>
      {posts.map(post => (
        <article key={post.id}>{post.title}</article>
      ))}
    </div>
  )
}

// Funziona così:
// 1. Al build → genera HTML statico
// 2. Per 1 ora → serve quello
// 3. Dopo 1 ora → rigenera in background
// 4. Visitatori vedono sempre pagine veloci!
🎯 ISR: Il Meglio dei Due Mondi

ISR ti dà performance di SSG con freschezza dei dati di SSR. La pagina è sempre veloce, ma i dati si aggiornano automaticamente!

Tabella di Confronto

Strategia Velocità Freschezza Dati Caso d'Uso
SSG ⚡⚡⚡ Massima ❌ Stale fino al rebuild Landing page, documentazione
ISR ⚡⚡ Molto alta ✅ Aggiornato ogni N sec Blog, e-commerce, news
SSR ⚡ Dipende dal server ✅✅ Sempre fresco Dashboard, dati real-time

Come Scegliere?

Fai queste domande:

  1. I dati cambiano?
    • Mai/raramente → SSG
    • Ogni ora/giorno → ISR
    • Continuamente → SSR
  2. I dati sono personalizzati per utente?
    • No → SSG o ISR
    • Sì → SSR o Client-Side
  3. La velocità è critica?
    • Sì → Preferisci SSG/ISR
    • No → SSR va bene

Esempio Pratico: Blog

// app/blog/page.js - Lista post
async function getPosts() {
  const res = await fetch('https://cms.example.com/posts', {
    next: { revalidate: 3600 } // ISR: revalida ogni ora
  })
  return res.json()
}

export default async function BlogList() {
  const posts = await getPosts()
  return (
    <div>
      {posts.map(post => (
        <Link key={post.id} href={`/blog/${post.slug}`}>
          {post.title}
        </Link>
      ))}
    </div>
  )
}

// app/blog/[slug]/page.js - Singolo post
async function getPost(slug) {
  const res = await fetch(`https://cms.example.com/posts/${slug}`, {
    next: { revalidate: 600 } // ISR: revalida ogni 10 min
  })
  return res.json()
}

export default async function BlogPost({ params }) {
  const post = await getPost(params.slug)
  return (
    <article>
      <h1>{post.title}</h1>
      <div dangerouslySetInnerHTML={{ __html: post.content }} />
    </article>
  )
}

// Genera pagine statiche per tutti i post al build
export async function generateStaticParams() {
  const posts = await fetch('https://cms.example.com/posts').then(r => r.json())
  
  return posts.map(post => ({
    slug: post.slug
  }))
}
🚀 generateStaticParams

Questa funzione dice a Next.js quali pagine dinamiche generare al build. Senza di essa, le pagine dinamiche sarebbero SSR!

Server Components

La Rivoluzione di React

I React Server Components (RSC) sono la feature più rivoluzionaria di Next.js 13+.

🎯 Concetto Chiave

Nell'App Router, ogni componente è un Server Component per default. Eseguono sul server, non inviano JavaScript al browser, e possono fare fetch direttamente.

Server Component = Default

// app/posts/page.js
// Questo è un SERVER COMPONENT (default in app/)

async function getPosts() {
  const res = await fetch('https://api.example.com/posts')
  return res.json()
}

export default async function Posts() {
  const posts = await getPosts()
  
  return (
    <div>
      {posts.map(post => (
        <article key={post.id}>
          <h2>{post.title}</h2>
        </article>
      ))}
    </div>
  )
}

// Questo componente:
// ✅ Esegue sul server
// ✅ Può essere async
// ✅ Può accedere direttamente al database
// ✅ Non invia JavaScript al browser
// ❌ NON può usare useState, useEffect, onClick, ecc.

Vantaggi dei Server Components

  • Zero JavaScript al client - bundle più piccolo
  • Accesso diretto ai dati - DB, filesystem, API private
  • Sicurezza - API keys e segreti restano sul server
  • Performance - elaborazione pesante sul server
  • Streaming - invii HTML man mano che si genera

Cosa Puoi Fare

// ✅ PUOI FARE nei Server Components:

// 1. Fetch dati
const data = await fetch('...')

// 2. Leggere dal database
import { db } from '@/lib/db'
const users = await db.users.findMany()

// 3. Leggere file system
import fs from 'fs/promises'
const content = await fs.readFile('data.json')

// 4. Usare variabili d'ambiente
const apiKey = process.env.SECRET_API_KEY

// 5. Importare librerie server-only
import { serialize } from 'server-only-lib'

Cosa NON Puoi Fare

// ❌ NON PUOI FARE nei Server Components:

// 1. Hooks di React
const [count, setCount] = useState(0)  // ❌ Errore!
useEffect(() => {}, [])                // ❌ Errore!

// 2. Event handlers
<button onClick={() => alert('hi')}>  // ❌ Errore!

// 3. Browser APIs
window.localStorage.setItem(...)       // ❌ Errore!
document.querySelector(...)            // ❌ Errore!

// 4. Context API
const value = useContext(MyContext)    // ❌ Errore!
⚠️ Quando Serve Interattività?

Se il componente ha bisogno di stato, eventi, o hooks → usa un Client Component (prossimo capitolo)

Accesso Diretto al Database

Uno dei superpotteri dei Server Components:

// lib/db.js
import { PrismaClient } from '@prisma/client'
export const db = new PrismaClient()

// app/users/page.js
import { db } from '@/lib/db'

export default async function Users() {
  // Fetch DIRETTO dal database!
  const users = await db.user.findMany({
    orderBy: { createdAt: 'desc' },
    take: 10
  })
  
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name} - {user.email}</li>
      ))}
    </ul>
  )
}

// Il database URL e le query NON vengono mai inviati al browser!

Composizione Server + Client

Il vero potere è combinare Server e Client Components:

// app/posts/page.js - SERVER COMPONENT
import PostCard from './PostCard'  // Client Component
import Comments from './Comments'  // Server Component

async function getPosts() {
  const res = await fetch('https://api.example.com/posts')
  return res.json()
}

export default async function Posts() {
  const posts = await getPosts()
  
  return (
    <div>
      {posts.map(post => (
        <div key={post.id}>
          {/* Dati fetchati sul server */}
          <Comments postId={post.id} />  {/* Server Component */}
          
          {/* UI interattiva nel browser */}
          <PostCard post={post} />        {/* Client Component */}
        </div>
      ))}
    </div>
  )
}

Streaming con Suspense

I Server Components supportano lo streaming - inviano HTML progressivamente:

// app/dashboard/page.js
import { Suspense } from 'react'
import Stats from './Stats'          // Fetch lento
import RecentOrders from './Orders'  // Fetch veloce

export default function Dashboard() {
  return (
    <div>
      <h1>Dashboard</h1>

      {/* Mostra fallback mentre carica */}

      </Suspense>

}

// 1. Header + RecentOrders → subito
// 2. "Caricamento statistiche..." → placeholder
// 3. Stats completo → quando pronto (streaming!)

Regole Importanti

📜 Le 3 Regole d'Oro
  1. Server Components possono importare Client Components ✅
  2. Client Components NON possono importare Server Components ❌
    (Ma possono riceverli come children o props)
  3. Props passati a Client Components devono essere serializzabili
    (No funzioni, Date objects vanno come string)
🎯 Esercizio: Server Component in Azione

Crea una pagina che:

  1. Fetcha utenti da https://jsonplaceholder.typicode.com/users
  2. Mostra lista utenti (Server Component)
  3. Ogni utente ha un bottone "Dettagli" interattivo (Client Component)
  4. Cliccando mostra/nasconde i dettagli dell'utente

Client Components

Quando Serve JavaScript nel Browser

I Client Components sono i componenti React "tradizionali" che conosci - eseguono nel browser e permettono interattività.

La Direttiva 'use client'

Per creare un Client Component, aggiungi 'use client' in cima al file:

// components/Counter.js
'use client'  // ← Questa direttiva è OBBLIGATORIA!

import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)
  
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>
        +1
      </button>
    </div>
  )
}
⚠️ Senza 'use client'

Se provi a usare useState, onClick, o altri hook in un componente senza 'use client', otterrai un errore! Next.js assume che tutti i componenti in app/ siano Server Components.

Quando Usare Client Components

Usa Client Components quando hai bisogno di:

  • Interattività - onClick, onChange, onSubmit
  • State - useState, useReducer
  • Effetti - useEffect, useLayoutEffect
  • Browser APIs - localStorage, window, document
  • Context - useContext, Context Provider
  • Librerie client-only - chart libraries, animation libs
  • Custom hooks - qualsiasi hook personalizzato

Esempi Comuni

Form Interattivo

// components/ContactForm.js
'use client'

import { useState } from 'react'

export default function ContactForm() {
  const [email, setEmail] = useState('')
  const [message, setMessage] = useState('')
  const [status, setStatus] = useState('idle')
  
  async function handleSubmit(e) {
    e.preventDefault()
    setStatus('loading')
    
    const res = await fetch('/api/contact', {
      method: 'POST',
      body: JSON.stringify({ email, message })
    })
    
    if (res.ok) {
      setStatus('success')
      setEmail('')
      setMessage('')
    } else {
      setStatus('error')
    }
  }
  
  return (
    <form onSubmit={handleSubmit}>
      <input 
        value={email}
        onChange={(e) => setEmail(e.target.value)}
        placeholder="Email"
      />
      <textarea
        value={message}
        onChange={(e) => setMessage(e.target.value)}
        placeholder="Messaggio"
      />
      <button disabled={status === 'loading'}>
        {status === 'loading' ? 'Invio...' : 'Invia'}
      </button>
      {status === 'success' && <p>✅ Inviato!</p>}
      {status === 'error' && <p>❌ Errore</p>}
    </form>
  )
}

Modal Interattivo

// components/Modal.js
'use client'

import { useState, useEffect } from 'react'

export default function Modal({ children, trigger }) {
  const [isOpen, setIsOpen] = useState(false)
  
  // Chiudi con ESC
  useEffect(() => {
    function handleEsc(e) {
      if (e.key === 'Escape') setIsOpen(false)
    }
    
    if (isOpen) {
      document.addEventListener('keydown', handleEsc)
      return () => document.removeEventListener('keydown', handleEsc)
    }
  }, [isOpen])
  
  return (
    <>
      <button onClick={() => setIsOpen(true)}>
        {trigger}
      </button>
      
      {isOpen && (
        <div 
          className="modal-overlay"
          onClick={() => setIsOpen(false)}
        >
          <div 
            className="modal-content"
            onClick={(e) => e.stopPropagation()}
          >
            {children}
            <button onClick={() => setIsOpen(false)}>
              Chiudi
            </button>
          </div>
        </div>
      )}
    </>
  )
}

Uso di Browser APIs

// components/ThemeSwitcher.js
'use client'

import { useState, useEffect } from 'react'

export default function ThemeSwitcher() {
  const [theme, setTheme] = useState('light')
  
  // Carica tema salvato
  useEffect(() => {
    const saved = localStorage.getItem('theme') || 'light'
    setTheme(saved)
    document.documentElement.setAttribute('data-theme', saved)
  }, [])
  
  function toggleTheme() {
    const newTheme = theme === 'light' ? 'dark' : 'light'
    setTheme(newTheme)
    localStorage.setItem('theme', newTheme)
    document.documentElement.setAttribute('data-theme', newTheme)
  }
  
  return (
    <button onClick={toggleTheme}>
      {theme === 'light' ? '🌙' : '☀️'}
    </button>
  )
}

Pattern: Composizione Smart

La chiave è minimizzare la quantità di codice client-side:

// ❌ MALE - Tutto è client
'use client'

async function getData() { ... }

export default function Page() {
  const [data, setData] = useState(null)
  
  useEffect(() => {
    getData().then(setData)
  }, [])
  
  return <div>{data?.map(...)}</div>
}

// ✅ BENE - Solo parte interattiva è client
// app/page.js - SERVER COMPONENT
async function getData() { ... }

export default async function Page() {
  const data = await getData()  // Fetch sul server
  
  return (
    <div>
      {data.map(item => (
        <ItemCard key={item.id} item={item} />  {/* Client */}
      ))}
    </div>
  )
}

// components/ItemCard.js - CLIENT COMPONENT
'use client'
export default function ItemCard({ item }) {
  const [liked, setLiked] = useState(false)
  return (
    <div>
      {item.name}
      <button onClick={() => setLiked(!liked)}>
        {liked ? '❤️' : '🤍'}
      </button>
    </div>
  )
}

Context Provider in Client

I Context Providers devono essere Client Components:

// app/providers.js
'use client'

import { createContext, useState } from 'react'

export const CartContext = createContext()

export function CartProvider({ children }) {
  const [items, setItems] = useState([])
  
  function addItem(item) {
    setItems([...items, item])
  }
  
  return (
    <CartContext.Provider value={{ items, addItem }}>
      {children}
    </CartContext.Provider>
  )
}

// app/layout.js - SERVER COMPONENT
import { CartProvider } from './providers'

export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <CartProvider>  {/* Client Component */}
          {children}      {/* Server Components dentro! */}
        </CartProvider>
      </body>
    </html>
  )
}

Metadata Dinamica

Per pagine dinamiche (come i post del blog), usa generateMetadata:

// app/blog/[slug]/page.js

// Funzione async che genera metadata
export async function generateMetadata({ params }) {
  // Fetcha dati del post
  const post = await fetch(`https://api.example.com/posts/${params.slug}`)
    .then(res => res.json())
  
  return {
    title: post.title,
    description: post.excerpt,
    
    // 🎯 CANONICAL - ecco come si fa!
    alternates: {
      canonical: `/blog/${params.slug}`
    },
    
    // Open Graph per social
    openGraph: {
      title: post.title,
      description: post.excerpt,
      type: 'article',
      publishedTime: post.publishedAt,
      authors: [post.author.name],
      images: [
        {
          url: post.coverImage,
          width: 1200,
          height: 630,
          alt: post.title,
        }
      ],
    },
    
    // Twitter Card
    twitter: {
      card: 'summary_large_image',
      title: post.title,
      description: post.excerpt,
      images: [post.coverImage],
    },
    
    // Altri meta tag
    robots: {
      index: true,
      follow: true,
    }
  }
}

// La pagina
export default async function BlogPost({ params }) {
